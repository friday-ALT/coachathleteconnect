2) The core features that must be implemented “as a system” so connections work reliably

For athlete–coach connections, reliability comes from three things: a correct state machine, transactional writes, and consistent authorization.

A. Connection state machine (non-negotiable)

Model connections with explicit statuses:

NONE (no relationship)

PENDING (request sent)

ACCEPTED

DECLINED

BLOCKED (optional, but recommended)

Rules:

Only one active connection record per athlete–coach pair (enforce with a unique index).

Only the recipient can accept/decline.

Either party can block; block overrides everything.

B. Transactional operations (prevents “half-created” bugs)

When a request is sent/accepted:

Write the connection row

Create notification row(s)

Optionally create a conversation thread
All in one transaction, so you never end up with “request exists but chat thread missing” or vice versa.

C. Authorization checks everywhere (prevents cross-role leaks)

Every endpoint should enforce:

Authenticated user exists

Their role matches required role

They’re part of that connection (or allowed to view public profile)
