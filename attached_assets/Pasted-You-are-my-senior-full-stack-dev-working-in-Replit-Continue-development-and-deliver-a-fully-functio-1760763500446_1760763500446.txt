## Core UX (make this work end-to-end)
When a user signs in, show a **split screen with two large tabs/cards** side-by-side:
- Left: **Athlete**
- Right: **Coach**
These are visually large, clickable, and responsive (stack on mobile).

### Auth / Entry
- Landing route `/` is a **Login** page (email, password, phone). Link to “Create account”.
- **Signup (Onboarding)** page `/onboarding` has a **top slider/tab** to choose **Athlete** or **Coach** and shows the appropriate form.

### Onboarding fields & persistence
- **Athlete**: email, phone number, age (int), skill level (Beginner/Intermediate/Advanced), location (city/state), password.
- **Coach**: name, location (city/state), short line of previous experience (≤160 chars), charging rate/price per hour (int), email, phone number, password, profile picture.
- Validate all fields; save to DB via Prisma. Passwords hashed.

### Dashboard behavior
Route: `/dashboard` (protected)
- Show the two large tabs/cards (**Athlete**, **Coach**). If a user has only one role, emphasize that card but keep both visible.
- Clicking **Athlete** opens the **Athlete view**:
  - A search bar + filters: `age`, `groupSize`, `skillLevel`.
  - Render a **paginated list of coaches** with: rating out of 5 (avg), profile picture, name, location, price/hour.
  - Each coach card has a **“Request Time Slot”** button → modal to submit: `groupSize`, `position`, optional `note`. Persist a TimeSlotRequest.
- Clicking **Coach** opens the **Coach view**:
  - Show a list/table of **incoming TimeSlotRequest** items with athlete **name, age, position, avatar, location, requested group size**.
  - Provide **Accept** and **Decline** actions to update request status.

### API routes (App Router)
- `POST /api/auth/register` (create user + profile)
- NextAuth Credentials at `/api/auth/[...nextauth]`
- `GET /api/coaches?skillLevel=&age=&groupSize=&q=` (indexed, paginated)
- `POST /api/requests` (athlete only)
- `PATCH /api/requests/:id` (coach only: accept/decline)
- `POST /api/reviews` (athlete only, after accepted request)
- Middleware to protect `/dashboard` and role guard on mutations.

### Data model (Prisma)
- User { id, email (unique), phone, passwordHash, role: 'ATHLETE' | 'COACH', createdAt }
- AthleteProfile { id, userId, age, skillLevel ('Beginner'|'Intermediate'|'Advanced'), locationCity, locationState }
- CoachProfile { id, userId, name, locationCity, locationState, experience, pricePerHour (int), ratingAvg (float, default 0), ratingCount (int, default 0), avatarUrl }
- TimeSlotRequest { id, athleteId (User), coachId (User), groupSize (int), desiredPosition (string), note (string?), status ('PENDING'|'ACCEPTED'|'DECLINED'), createdAt }
- Review { id, athleteId, coachId, rating (1–5), comment (string?), createdAt }

Run migrations and include a small **seed script** (2–3 coaches, 1–2 athletes).
